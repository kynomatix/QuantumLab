Build: Flux Momentum Backtester and Optimizer with UI
Overview
Build a web application that lets me paste in a Pine Script trading strategy, select a ticker and timeframe, and run a local parameter optimization backtest against Binance USD-M futures OHLCV data. The app should display results in a clean, professional dashboard.
Do NOT use Streamlit. Use a proper React frontend (Next.js or Vite + React) with Tailwind CSS and a Python FastAPI backend. The UI should look professional, modern, and dark-themed (trading app aesthetic).

Pine Script Parser Requirements
The parser needs to extract input declarations from Pine Script. Here is the pattern:
python# Parse lines like:
# int bbLen = input.int(20, "BB Length", minval=5, maxval=50, group=g_squeeze)
# float bbMult = input.float(2.0, "BB Multiplier", minval=0.5, maxval=4.0, step=0.1, group=g_squeeze)
# bool useHullFilter = input.bool(true, "Enable Hull Filter", group=g_trend)
# string tradeDir = input.string("Both", "Trade Direction", options=["Both", "Long Only", "Short Only"], group=g_dir)
Extract: variable name, type, default value, label/title, min, max, step, group name, options list.
Also parse group variable declarations like:
string g_squeeze = "═══ SQUEEZE DETECTION ═══"
Map group variables to their display names and use them to organize the parameter display.
For the actual strategy logic conversion (Pine to Python backtest engine), the system needs to handle:
Entry Logic Patterns
Look for strategy.entry calls and trace back to find the conditions. Common patterns:

Long entry: if longCondition ... strategy.entry("Long", strategy.long)
Short entry: if shortCondition ... strategy.entry("Short", strategy.short)
Extract the boolean conditions that gate entries

Exit Logic Patterns

strategy.exit calls with stop, limit, trail_points, trail_offset, qty_percent parameters
strategy.close calls for signal-based exits
Multiple strategy.exit calls for multi-tier TP (TP1, TP2, TP3 with different qty_percent values)

Indicator Patterns
Common indicators to detect and implement in Python:

SMA, EMA, WMA, Hull MA
Bollinger Bands (basis ± mult * stdev)
Keltner Channel (EMA ± mult * ATR)
RSI, ADX, ATR
Linear regression (linreg)
Volume comparisons (volume vs SMA of volume)
Squeeze detection (BB inside KC)

Important Implementation Notes for the Python Backtester

Process bars sequentially (bar by bar, no lookahead)
Entry price = bar close (simulating next bar open)
Stop loss and take profit checks use bar high and low (intrabar fills)
Conditional/signal exits use bar close price
Commission applied both entry and exit (default 0.05% each way)
Position sizing: fixed cash amount per trade (default $1000 with $100 capital = 10x simulation)
Pyramiding = 0 (one position at a time)
Partial take profits: each TP level closes a percentage of the current remaining position
Trailing stop ratchets only in the profit direction (max for longs, min for shorts)
Breakeven moves stop to entry + small offset after a TP is hit

Parameter Optimization Engine

Random search: sample N random parameter vectors within the parsed bounds
For int params: uniform random between min and max
For float params: snap to step size
For bool params: random True/False
For string params: random choice from options list
Neighbourhood search: jitter 2-5 parameters by ~15% of their range
Enforce logical constraints (e.g. TP1 < TP2 < TP3)
Ranking: Net Profit % (primary descending), Win Rate % (secondary descending), Max Drawdown % (tertiary ascending)
Filters: minimum trades threshold, max drawdown soft penalty

Important Notes

The Pine Script parser does NOT need to perfectly convert ALL Pine logic to Python automatically. It primarily needs to extract the INPUT PARAMETERS reliably. For the strategy logic conversion, it should handle common patterns (indicators, entry/exit conditions, position management) and the user can verify/adjust. The key value is the parameter extraction and optimization loop.
The backtester must be deterministic. Same inputs = same outputs every time. Use seeded random for optimization.
Cache OHLCV data aggressively. Fetching is slow, computation is fast. First run per ticker/timeframe fetches and caches. All subsequent optimization samples reuse cached data.
The UI must show real-time progress. Long optimization runs (900 random + 1200 refinements) can take minutes per ticker/timeframe. Users need to see it's working and how far along it is.
All monetary calculations use: $100 initial capital, $1000 per trade (simulating 10x leverage), 0.05% commission each way.
Make the "Copy Best Config" export format the parameters as Pine Script input overrides so the user can paste them directly back into TradingView to verify the results visually.
